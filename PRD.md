## 1. 背景与目标

### 1.1 背景

健康宣教系统每天向患者推送大量病种相关的健康宣教内容，但运营和医院管理者无法直观掌握：
- 每个医院、每个病种的推送情况
- 患者实际查看宣教内容的情况
- 哪些病种的宣教内容更受患者关注、哪些被忽视

---

## 2. 范围

### 2.1 本期范围（In Scope）

- 顶部医院切换下拉菜单（支持多家医院）
- 4 个核心指标卡片：累计推送次数、今日推送次数、累计查看次数、综合查看率
- 病种推送与查看（历史总量）柱状图
- 病种详情（当天数据）卡片列表，支持按日期筛选
- "查看详情"和"查看全部"按数据范围展开完整明细表
- 单个病种卡片点击查看该病种详情（含当日数据 + 历史总量）


## 3. 功能需求

### 3.1 页面整体布局

页面分为三个区域：

| 区域 | 内容 |
|------|------|
| 顶部 | 医院切换下拉菜单 + 系统标题 + 实时时间 |
| 中部 | 4 个并列的核心指标卡片 |
| 主区 | 左侧"病种推送与查看（历史总量）"柱状图 + 右侧"病种详情（当天数据）"卡片列表 |



### 3.2 医院切换功能

#### 3.2.1 功能描述

页面顶部左侧提供下拉菜单，用户可在多家医院间切换。切换后，页面所有数据立即更新为该医院的数据。

#### 3.2.2 主流程

1. 用户进入页面，默认加载第一家医院的数据
2. 用户点击下拉菜单，展开可选医院列表
3. 页面所有模块（指标卡 / 柱状图 / 病种列表）刷新为该医院的数据

#### 3.2.3 业务规则
- 不同医院的病种配置不同（病种来源于后台规则库，与医院绑定，，医院取规则配置中的医院数据）

#### 3.2.4 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 医院列表为空 | 顶部下拉菜单灰色禁用，主区显示"暂无可访问医院" |
| 切换后数据加载失败 | 弹出提示"数据加载失败，请重试"，保留上一次的数据 |
| 切换后该医院无病种 | 柱状图和病种列表显示"该医院暂无病种配置" |

---

### 3.3 核心指标卡片

#### 3.3.1 功能描述

页面中部展示 4 个并列的指标卡片，反映当前医院的关键数据。

#### 3.3.2 指标清单

| 指标名 | 数据范围 | 说明 |
|--------|---------|------|
| 累计推送次数 | 历史总量 | 当前医院所有病种历史推送总次数 |
| 今日推送次数 | 当日推送总量
| 累计查看次数 | 历史总量 | 当前医院所有病种历史被查看总次数 |
| 综合查看率 | 历史总量 | 累计查看次数 / 累计推送次数，保留 2 位小数 |

#### 3.3.3 业务规则
- 卡片数据随医院切换实时更新
- 综合查看率显示百分比格式（如 `61.41%`）
- 数据为 0 时正常显示 `0` 或 `0%`，不显示为空

---

### 3.4 病种推送与查看（历史总量）

#### 3.4.1 功能描述

主区左侧展示柱状图，**展示当前医院各病种的历史总量**（累计推送 vs. 累计查看），横轴为病种名称，纵轴为次数，每个病种并列显示"推送次数"和"查看次数"两根柱子。

#### 3.4.2 业务规则

- 数据范围：历史总量，**不受右侧日期筛选影响**
- 横轴病种顺序：展示推送量前9的数据
- 鼠标悬停柱子显示该病种的精确数字
- 病种名称过长时旋转 25° 显示

#### 3.4.3 查看详情

- 卡片右上角"查看详情"按钮，点击后弹出**历史总量明细弹窗**（详见 3.6.1）

---

### 3.5 病种详情（当天数据）

#### 3.5.1 功能描述

主区右侧以卡片网格形式展示当前医院的所有病种，每张卡片显示该病种**在所选日期当天**的推送次数、查看次数和**平均观看时长**（按查看次数加权计算）。

#### 3.5.2 日期筛选器

- 卡片标题右侧提供日期选择器，默认值为今日
- 日期选择器最大值为今日，不允许选择未来日期
- 选择某个日期后：
  - 所有病种卡片的推送/查看数字立即更新为该日数据
  - 顶部"今日推送次数"卡片同步变为该日推送总数，标签变为"YYYY-MM-DD 推送次数"
  - 选回今日时，标签恢复为"今日推送次数"

#### 3.5.3 业务规则

- 病种来源于后台规则库，按医院隔离
- 卡片数据范围为「所选日期当天」，从 0 点到 24 点
- 病种卡片点击后弹出该病种详情（详见 3.7 节）
- 病种较多时卡片区域支持垂直滚动

#### 3.5.4 查看全部

- 卡片右上角"查看全部"按钮，点击后弹出**当日明细弹窗**（详见 3.6.2）

#### 3.5.5 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 所选日期当天无任何推送/查看 | 卡片数字均显示 `0`，查看率显示 `0.0%` |
| 切换日期后接口失败 | 弹出"日期数据加载失败"，保留上一次数据 |
| 数据加载失败时，显示重试按钮

---

### 3.6 明细弹窗

弹窗按数据范围分为两类，结构一致，差异仅在数据来源和标题。

#### 3.6.1 历史总量明细弹窗（柱状图 → 查看详情）

- 标题：「病种推送与查看 - 历史总量明细」
- 顶部标签：「数据范围：历史总量」
- 汇总卡片：累计推送（蓝）、累计查看（绿）、综合查看率（橙）
- 明细表：列出当前医院所有病种的**历史总量**（推送 / 查看 / 查看率 / **平均观看时长**），末行为合计（平均观看时长按查看次数加权计算）
- 每行病种均**可展开**，展开后显示该病种下所有规则节点的明细（详见 3.6.3）

#### 3.6.2 当日明细弹窗（病种详情 → 查看全部）

- 标题：「病种详情 - YYYY-MM-DD 当日明细」
- 顶部标签：「数据范围：当日（YYYY-MM-DD）」
- 汇总卡片：当日推送、当日查看、当日查看率
- 明细表：列出当前医院所有病种在**所选日期**当天的数据（推送 / 查看 / 查看率 / **平均观看时长**），末行为合计（平均观看时长按查看次数加权计算）
- 每行病种均**可展开**，展开后显示该病种下所有规则节点的明细（详见 3.6.3）

#### 3.6.3 病种行展开 - 规则节点下钻

**交互**
- 表格每行病种最左侧显示展开/折叠箭头（▶ / ▼）
- 点击病种行任意位置可展开或收起该行
- 多个病种行可同时展开，互不影响

**展开后内容**
- 标签：「规则节点明细 - 历史总量」或「规则节点明细 - 当日（YYYY-MM-DD）」 + 「共 N 条规则」
- 子表格列：规则名称、推送次数、查看次数、查看率
- 末行为该病种规则节点的合计（应等于父行的病种数据）

**业务规则**
- 展开数据的时间维度与所在弹窗保持一致（历史弹窗展开看历史节点数据，当日弹窗展开看当日节点数据）
- 规则节点的来源、命名同 3.7.3 节
- 节点合计 = 父行病种数据，验证数据准确性

#### 3.6.4 通用业务规则

- 表格按总推送量顺序排列
- 查看率精确到 2 位小数；推送为 0 时查看率显示 `0.0%`
- 点击弹窗外区域或右上角"×"关闭弹窗

---

### 3.7 单病种详情弹窗

#### 3.7.1 功能描述

用户点击病种详情列表中某张病种卡片时，弹出该病种的详细信息，**重点展示该病种下所有规则节点的推送 / 查看明细**，同时对比当日数据与历史总量。

> 业务背景：每个病种在规则库中通常对应多条规则（即"节点"），例如"屈光手术"病种下可能包含「术前评估提醒」「术前检查通知」「术后第1日复查」等多个规则节点。每个节点都有独立的推送与查看数据，需要在病种详情中下钻展示。

#### 3.7.2 弹窗内容结构

弹窗自顶向下分为四块：

1. **基本信息**
   - 病种描述

2. **规则节点明细 - 当日（YYYY-MM-DD）**
   - 顶部统计行：总规则数 | 当日推送 | 当日查看 | 当日查看率
   - 明细表：列出该病种下所有规则节点
     - 列：规则名称、推送次数、查看次数、查看率、**平均观看时长**
     - 末行为合计（平均观看时长按"按查看次数加权"计算）
   - 数据范围：所选日期当天

3. **规则节点明细 - 历史总量**
   - 顶部统计行：总规则数 | 累计推送 | 累计查看 | 综合查看率
   - 明细表结构同上（包含"平均观看时长"列）
   - 数据范围：历史总量

#### 3.7.3 数据来源

- **规则节点列表**：来源于后台「病种规则库」，每个病种 → 多条规则的多对一关系
- **节点统计**：按"医院 + 病种 + 规则节点 + 时间维度"聚合推送和查看记录

#### 3.7.4 业务规则

- 规则节点的名称、顺序与后台规则库保持一致
- 规则节点的总和（推送 / 查看）= 该病种卡片显示的数字
- 风险等级根据级别显示不同颜色：高（红）/ 中等（橙）/ 低（绿）
- 基本信息内容由后台病种规则库提供
- "当日数据"中的日期跟随右侧日期筛选器
- 关闭逻辑同 3.6.4

#### 3.7.5 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 该病种暂无规则节点配置 | 显示"该病种暂未配置规则节点" |
| 当日所有规则节点推送均为 0 | 表格正常显示，数值显示 `0`，查看率显示 `0.0%` |

## 4. 数据需求说明

### 4.1 数据来源

大屏所需数据全部来自健康宣教系统后端。

- **推送数据**：来自用户端的推送埋点上报（详见 6.1.1）
- **查看数据**：以**用户端页面曝光**为准，由曝光埋点上报（详见 6.1.2）
- 字段定义、接口设计、聚合方式由开发团队负责

### 4.2 数据隔离原则

- 所有数据按医院维度隔离，前端始终携带当前医院标识获取数据
- 不同医院之间的数据相互不可见
- 病种数据与医院绑定，不同医院可配置不同的病种

### 4.3 病种与规则的层级关系

- 后台规则库结构为：**医院 → 病种 → 规则节点**（三级结构）
- 一个病种下可包含 1 到 N 个规则节点（如「屈光手术」下的「术前检查通知」「术后第1日复查」等）
- 推送和查看的最小统计粒度为「规则节点」，向上聚合得到「病种」、再向上聚合得到「医院」


### 4.5 数据更新

- 进入页面或切换医院时获取最新数据（含历史总量 + 当日数据）
- 切换日期时仅刷新与该日相关的数据
- 页面支持数据5分钟自动刷新，展示动态变化效果

## 6. 埋点需求

埋点分为两部分：**用户端业务埋点**（提供大屏的源数据）和 **大屏前端行为埋点**（统计大屏自身使用情况）。

### 6.1 用户端业务埋点（数据来源）

大屏所有推送 / 查看数据均来源于用户端（患者端）的埋点上报。**所有埋点均按医院 + 病种 + 规则节点三级维度记录**，便于后台按任意层级聚合统计。

#### 6.1.1 推送埋点

- **事件名（建议）**：`edu_push_sent`
- **触发时机**：每次向用户端实际发出一条宣教推送时上报（一次推送 = 一条记录）
- **统计粒度（从细到粗）**：
  - 每个规则节点的推送次数（最小粒度）
  - 每个病种的推送次数（= 该病种下所有规则节点之和）
  - 每个医院的推送次数（= 该医院下所有病种之和）
- **必报字段**：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `push_id` | string | 推送唯一 ID，全局唯一，用于与查看埋点关联 | `psh_20260526_0001` |
| `hospital_id` | string | 医院 ID（聚合维度1） | `H001` |
| `hospital_name` | string | 医院名称（冗余便于查询） | `衢州市人民医院` |
| `disease_id` | string | 病种 ID（聚合维度2） | `D_QGSS` |
| `disease_name` | string | 病种名称 | `屈光手术` |
| `rule_id` | string | 规则节点 ID（聚合维度3，最小粒度） | `R_QGSS_PRE_01` |
| `rule_name` | string | 规则节点名称 | `术前评估提醒` |
| `patient_id` | string | 患者匿名 ID | `u_xxxxxxxx` |
| `push_channel` | enum | 推送渠道：`app` / `sms` / `wechat` 等 | `app` |
| `push_time` | datetime | 推送时间（ISO8601，精确到秒） | `2026-05-26T10:23:11+08:00` |
| `client_version` | string | 用户端版本号 | `2.3.1` |

- **聚合方式**：后台按时间维度（按日 / 历史总量）+ 三级业务维度（医院 / 病种 / 规则节点）聚合：
  - 大屏「历史总量推送数」= `COUNT(push_id) GROUP BY hospital_id`
  - 大屏「病种推送数」= `COUNT(push_id) GROUP BY hospital_id, disease_id`
  - 弹窗「规则节点推送数」= `COUNT(push_id) GROUP BY hospital_id, disease_id, rule_id`
- **数据保留**：明细数据保留 ≥ 2 年，按日聚合数据长期保留

#### 6.1.2 查看埋点

- **事件名（建议）**：`edu_page_exposure`
- **查看定义**：以**页面曝光**为准——用户端宣教内容页面实际渲染并展示给用户即记为一次"查看"
- **触发时机**：页面曝光开始时记录开始时间；页面关闭 / 离开 / 切到后台时上报曝光时长
- **必报字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `push_id` | string | 关联的推送 ID（用于反查 医院 / 病种 / 规则节点） |
| `hospital_id` / `disease_id` / `rule_id` | string | 同推送埋点（冗余下发，便于直接聚合） |
| `patient_id` | string | 患者匿名 ID |
| `exposure_start` | datetime | 曝光开始时间 |
| `exposure_end` | datetime | 曝光结束时间 |
| `duration_ms` | int | 曝光时长（毫秒，用于"平均观看时长"派生指标） |
| `exit_type` | enum | 退出方式：`close` / `back` / `background` / `crash` |

- **统计粒度**：与推送埋点保持一致（医院 / 病种 / 规则节点三级）
- **派生指标**：
  - 查看次数 = `COUNT(*)`（按对应粒度 GROUP BY）
  - 查看率 = 查看次数 / 推送次数
  - **平均观看时长 = `SUM(duration_ms) / COUNT(*)`**（按对应粒度 GROUP BY，单位毫秒，前端按需格式化为"X分Y秒"）

#### 6.1.3 业务规则

- 同一用户对同一内容的多次曝光均计入查看次数（重复曝光也算）
- 曝光时长 < 设定阈值（如 1 秒）的曝光可视为无效（具体阈值由产品和数据团队约定）
- 异常退出（崩溃 / 网络断开）时，应在下次启动时补报曝光时长
- 字段定义、数据结构、接口规范由开发团队设计
